Accessibility statement for Appsembler

This accessibility statement applies to the websites at: https://openedx.open.ac.uk

These websites are run by Appsembler. We want as many people as possible to be able to use these websites. For example, that means you should be able to:

  • change colours, contrast levels and fonts
  • navigate most of the website using just a keyboard
  • navigate most of the website using speech recognition software
  • listen to most of the website using a screen reader (including the most recent versions of JAWS, NVDA and VoiceOver)

    • We’ve also made the website text as simple as possible to understand.

    AbilityNet has advice on making your device easier to use if you have a disability.

    How accessible this website is

    We know some parts of this website are not fully accessible:

  • A visible tab focus indicator is not present on all pages
  • Alternative attributes are not always meaningful
  • Some copy has a low colour contrast to the site background
  • Some form inputs do not have associated labels within the HTML
  • Autocomplete attributes have not been applied to form input fields
  • Text does not reflow to a single column at 400% zoom
  • Boundaries of input fields have a low contrast

    • Feedback and contact information

    If you need information on this website in a different format like accessible PDF, large print, easy read, audio recording or braille:

  • accessibility-feedback@open.ac.uk
  • call 0300 303 5303
    • We’ll consider your request and get back to you in 10 days.

    Reporting accessibility problems with this website

    We’re always looking to improve the accessibility of this website. If you find any problems not listed on this page or think we’re not meeting accessibility requirements, contact: accessibility-feedback@open.ac.uk

    Enforcement procedure

    The Equality and Human Rights Commission (EHRC) is responsible for enforcing the Public Sector Bodies (Websites and Mobile Applications) (No. 2) Accessibility Regulations 2018 (the ‘accessibility regulations’). If you’re not happy with how we respond to your complaint, contact the Equality Advisory and Support Service (EASS).

    Technical information about this website’s accessibility

    Appsembler is committed to making its website accessible, in accordance with the Public Sector Bodies (Websites and Mobile Applications) (No. 2) Accessibility Regulations 2018.

    Compliance status

    This website is partially compliant with the Web Content Accessibility Guidelines version 2.1 AA standard, due to the non-compliances and exemptions listed below.

    Non-accessible content

    The content listed below is non-accessible for the following reasons.


    Non-compliance with the accessibility regulations

    A visual focus indicator is not always present when navigating the application with a keyboard. This is a failure of WCAG 2.1 success criterion 2.4.7 Focus Visible. We plan to add a consistent focus indicator by 1 July, 2021. Heading levels have been skipped on some pages of the application, these include ‘Profile’, ‘Course’ and ‘Dashboard’. This is a failure of WCAG 2.1 success criterion 2.4.1 Bypass Blocks. We plan to review and update the heading structure across the site by 1 July, 2021. Some images have incorrect alternative attributes. This is a failure of WCAG 2.1 success criterion 1.1.1 Non-Text Content. We plan to update the alternative attributes of images on the website by 1 July, 2021. Some copy has a low colour contrast to the site background, this is particularly notable on focused links. This is a failure of WCAG 2.1 success criterion 1.4.3 Contrast Minimum. We plan to update the colour contrast on the site by 1 July, 2021. On the ‘Discussion’ page the ‘Receive updates’ check box has an accessible name of Toggle Notification Settings which does not match the visible name. This is a failure of WCAG 2.1 success criterion 2.5.3 Label in Name. We plan to update the accessible name of this check box by 1 July, 2021. Boundaries of inputs do not have sufficient contrast in an unfocused state. This is present on the ‘Create an account’, ‘Sign in’, ‘Course’ and ‘Discussion’ pages. This is a failure of WCAG 2.1 success criterion 1.4.11 Non-text Contrast. We plan to update the contrast of input boundaries by 1 July, 2021. No detailed error suggestions are provided for the ‘email’ field on the Create an account’, ‘Sign in’ and ‘Account Settings’ pages. This is a failure of WCAG 2.1 success criterion 3.3.3 Error Suggestion. We plan to ensure that error suggestions are more detailed by 1 July, 2021.


    Disproportionate burden

    We have assessed the cost of fixing the issues with form elements not being grouped. We believe that doing so now would be a disproportionate burden within the meaning of the accessibility regulations. We will make another assessment when we have reviewed the budget for our next development round by the end of June 2021. We have assessed the cost of fixing the issues with at 200% text only zoom some text displays over other text or becomes cropped by the boundaries of components. This is a failure of WCAG 2.1 success criterion 1.4.4 Resize Text. We believe that doing so now would be a disproportionate burden within the meaning of the accessibility regulations. We will make another assessment when we have reviewed the budget for our next development round by the end of June 2021. We have assessed the cost of fixing the issues with at 400% zoom at a 1280px viewport two-dimensional scrolling is present. This is a failure of WCAG 2.1 success criterion 1.4.10 Reflow. We believe that doing so now would be a disproportionate burden within the meaning of the accessibility regulations. We will make another assessment when we have reviewed the budget for our next development round by the end of June 2021. We have assessed the cost of fixing the issue with no search or site map being present. This is a failure of WCAG 2.1 success criterion 2.4. Multiple Ways. We believe that doing so now would be a disproportionate burden within the meaning of the accessibility regulations. We will make another assessment when we have reviewed the budget for our next development round by the end of June 2021. We have assessed the cost of fixing the issue with form input fields not utilising autocomplete attributes. This is a failure of WCAG 2.1 success criterion 1.3.5 Identify Input Purpose. We believe that doing so now would be a disproportionate burden within the meaning of the accessibility regulations. We will make another assessment when we have reviewed the budget for our next development round by the end of June 2021.

    Content that’s not within the scope of the accessibility regulations

    User create content

    Any user created content such as videos, links, images and downloadable documents are outside of Appsembler’s control. Please directly contact the Module Team should any issues with these be present.
